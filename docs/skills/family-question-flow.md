# 가족 질문 만들기 플로우

## 기능 목적

가족 연결을 마친 사용자가 홈 화면에서 "질문 만들기"를 눌러 연결된 가족에게 질문을 보내는 플로우를 구현한다. 이 기능은 단순한 정적 화면이 아니라 현재 로그인한 사용자, 연결된 가족, DB에 등록된 질문 테마와 추천 질문을 기준으로 동작해야 한다.

바이브코딩 대회 발표에서는 아래 기준을 근거로 설명한다.

- 사용자가 실제로 연결된 가족에게만 질문을 보낼 수 있게 제한한다.
- 질문 테마와 추천 질문은 DB 조회 결과를 UI에 반영한다.
- 하드코딩 데이터는 API 실패나 초기 개발용 fallback에만 사용한다.
- 직접 입력과 추천 질문 선택이 충돌하지 않도록 질문 작성 상태를 명확히 관리한다.

## 사용자 흐름

1. 가족 연결을 완료한다.
2. 홈 화면에 진입한다.
3. 홈 화면의 "질문 만들기"를 클릭한다.
4. 질문 만들기 화면에 진입한다.
5. 받는 사람을 1명 선택한다.
6. 질문 테마를 1개 선택한다.
7. 추천 질문을 1개 선택하거나 직접 질문을 입력한다.
8. 질문 보내기 버튼으로 질문을 전송한다.
9. 전송 성공 후 홈 또는 질문&답변 화면으로 이동한다.

## 화면 구성 기준

### 홈 화면

- 기존 홈 화면 UI와 레이아웃을 유지한다.
- "질문 만들기" CTA만 필요한 범위에서 연결한다.
- CTA 클릭 시 질문 만들기 라우트로 이동한다.
- 전체 앱 레이아웃, `global.css`, root layout은 임의로 수정하지 않는다.

### 질문 만들기 화면

- 첨부된 디자인을 기준으로 구현한다.
- 기존 `src/components/ui` 디자인 시스템 컴포넌트를 우선 사용한다.
  - `Button`
  - `Avatar`
  - `Badge`
  - `Card`
  - `Input`
  - `Textarea`
  - `Toast`
- 색상, 타이포그래피, 간격은 `src/styles/tokens`의 디자인 토큰을 우선 사용한다.
- 하단 탭바는 기존 홈 화면과 동일한 스타일을 유지한다.
- 화면 추가와 API 연결에 필요한 변경만 수행하고, 앱 전체 UI를 새로 갈아엎지 않는다.

## DB/API 연결 기준

### 받는 사람 조회

- 받는 사람 목록은 하드코딩하지 않는다.
- 현재 로그인한 사용자와 연결된 가족만 DB/API에서 조회해 보여준다.
- 가족 연결 이후의 사용자만 진입하는 플로우지만, 응답이 비어 있을 수 있으므로 빈 상태를 처리한다.
- 표시 데이터는 API 응답 필드에 맞춘다.
  - 예: 가족 구성원 id, 이름, 역할, 프로필 이미지, 가족 id

### 질문 테마 기준

- 제품 기준 기본 테마는 `일상`, `추억`, `고민` 3개다.
- 현재 Question API 스펙에는 별도 theme 조회 엔드포인트가 없고 `depth=tiny|medium|deep` enum을 사용한다.
- 따라서 UI의 `일상`, `추억`, `고민`은 각각 `tiny`, `medium`, `deep`으로 매핑한다.
- 추후 DB theme 테이블/API가 추가되면 이 매핑을 API 조회 기반으로 교체한다.

### 추천 질문 조회

- 추천 질문은 선택된 질문 테마의 depth를 기준으로 DB/API에서 조회한다.
- 받는 사람 또는 테마가 선택되지 않은 상태에서는 추천 질문 조회를 하지 않거나 비활성 상태로 둔다.
- 선택값이 바뀌면 이전 추천 질문 선택은 초기화한다.
- 추천 질문이 없는 경우 직접 질문 입력을 유도한다.

### 질문 전송

질문 전송 시에는 최소한 아래 값을 저장한다.

- 받는 사람 id
- 질문 테마에 매핑된 depth
- 질문 내용
- 질문 유형
  - 추천 질문 선택: `recommendation`
  - 직접 질문 입력: `custom`

전송 API가 확정되지 않은 경우, 구현 시 `docs/route-map.md`의 미확인 항목을 갱신하고 API 클라이언트에는 임시 타입을 명확히 둔다.

## 상태 관리 기준

- 받는 사람은 1명만 선택할 수 있다.
- 테마는 1개만 선택할 수 있다.
- 추천 질문은 1개만 선택할 수 있다.
- 직접 질문 입력 시 추천 질문 선택과 충돌하지 않게 처리한다.
  - 추천 질문을 선택하면 직접 입력값을 비우거나 비활성화한다.
  - 직접 질문을 입력하면 선택된 추천 질문을 해제한다.
- 질문 내용이 없으면 보내기 버튼을 비활성화한다.
- 받는 사람 또는 테마가 없으면 보내기 버튼을 비활성화한다.
- 전송 중에는 중복 제출을 막기 위해 보내기 버튼을 비활성화하고 loading 상태를 표시한다.

## 예외 처리 기준

### 연결된 가족 없음

- 받는 사람 영역에 안내 문구를 표시한다.
- 질문 보내기 버튼은 비활성화한다.
- 필요하면 가족 연결 화면으로 이동할 수 있는 보조 CTA를 제공하되 기존 가족 연결 로직은 수정하지 않는다.

### 질문 테마 매핑

- 현재는 API enum 매핑이므로 네트워크 실패 케이스가 없다.
- 추후 theme 조회 API가 생기면 조회 실패 시 `일상`, `추억`, `고민` 기본 테마로 fallback한다.

### 추천 질문 없음

- 빈 목록을 그대로 방치하지 않는다.
- "직접 질문을 작성해 보세요" 같은 안내를 표시한다.
- 직접 질문 입력 영역은 계속 사용할 수 있어야 한다.

### 질문 전송 실패

- 화면 안에 에러 메시지 또는 toast를 표시한다.
- 사용자가 내용을 잃지 않고 다시 전송할 수 있어야 한다.
- 실패 시 선택한 받는 사람, 테마, 질문 내용은 유지한다.

### 질문 전송 성공

- 홈 또는 질문&답변 화면으로 이동한다.
- 이동 기준은 제품 플로우와 라우트맵을 따른다.
  - 홈: `/`
  - 질문&답변: `/questions`
- 성공 메시지가 필요하면 기존 `Toast` 패턴을 사용한다.

## 구현 시 주의사항

- 기존 가족 연결 로직을 깨뜨리지 않는다.
- 카카오톡 초대 기능을 수정하지 않는다.
- 초대 코드 연결 기능을 수정하지 않는다.
- role/onboarding 화면을 수정하지 않는다.
- 전체 UI를 새로 갈아엎지 않고, 질문 만들기 화면과 필요한 API 연결만 추가한다.
- 하드코딩 데이터는 임시 fallback에만 사용한다.
- 기본 사용자 흐름은 반드시 DB/API 조회 기반으로 구현한다.
- API 스펙이 불확실한 경우 임의 확정하지 말고 `docs/route-map.md`의 확인 필요 항목을 갱신한다.

## 구현 체크리스트

- 홈 화면 "질문 만들기" CTA에서 질문 만들기 화면으로 이동한다.
- 질문 만들기 라우트를 추가한다.
- 현재 사용자와 연결된 가족 조회 API를 연결한다.
- 질문 테마 UI를 API depth enum과 연결한다.
- 테마 선택값으로 추천 질문 조회 API를 연결한다.
- 추천 질문 선택과 직접 입력의 상호 배타 상태를 구현한다.
- 질문 내용, 받는 사람, 테마가 모두 유효할 때만 보내기 버튼을 활성화한다.
- 질문 전송 API를 연결한다.
- 전송 성공/실패 상태를 UI에 반영한다.
- 하단 탭바와 기존 앱 레이아웃을 유지한다.

## 질문&답변 탭 플로우

### 사용자 흐름

1. 하단 탭바에서 `질문&답변`을 누른다.
2. 받은 질문 목록(`/questions`)에 진입한다.
3. 현재 로그인한 사용자에게 온 질문만 확인한다.
4. 질문 카드를 누르면 받은 질문 상세(`/questions/[questionSendId]`)로 이동한다.
5. 상세에서 질문 내용, 보낸 사람 관계/이름, 답변 팁, 프라이버시 안내를 확인한다.
6. `질문에 답변하기`를 누르면 기존 영상 답변 플로우(`/questions/[questionSendId]/record`)로 이동한다.

### API 연결 기준

- 목록은 `GET /api/v1/answers/questions?unansweredOnly=false&sort=unanswered_first`를 사용한다.
- 상세는 URL의 `questionSendId`로 `GET /api/v1/answers/questions/{questionSendId}`를 호출한다.
- 상세 진입 시 읽지 않은 질문이면 `PATCH /api/v1/answers/questions/{questionSendId}/read`로 읽음 처리한다.
- 백엔드는 현재 사용자가 receiver가 아닌 질문을 404로 막아야 하며, 프론트는 이를 접근 불가/없음 메시지로 표시한다.
- sender의 `role`, `displayName`, `profileImageUrl`은 API 응답을 그대로 사용한다.

### UI 상태 기준

- 목록에는 sender 관계/이름, 질문 내용, 상태, 받은 시간을 표시한다.
- 상태는 `sent`/미답변이면 `답변 대기`, `answered`면 `답변 완료`, 그 외 `cancelled`/`expired`를 구분해 표시한다.
- 질문이 없으면 `아직 받은 질문이 없어요.`를 표시한다.
- 목록과 상세 모두 loading/error 상태를 표시한다.
- 하단 탭바의 현재 탭은 `질문&답변`으로 유지한다.

### 구현 주의사항

- 질문&답변 탭은 상세로 바로 가지 않고 항상 목록(`/questions`)을 먼저 보여준다.
- 상세 화면은 받은 질문 데이터만 보여주고, 질문 생성 화면의 추천/직접 입력 상태를 건드리지 않는다.
- `질문에 답변하기` CTA는 이미 구현된 영상 답변/업로드 플로우로 연결한다.
- 가족 연결, 온보딩, 카카오톡 초대, 코드 연결 로직은 수정하지 않는다.
